## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser/client -> /api/autentikasi/* | Untrusted email/password/nama input crosses here |
| client-supplied JWT -> protected endpoints | Token authenticity must be verified server-side |
| registration request body -> peran | Client must NOT be able to assign its own role |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Information Disclosure | password storage | mitigate | bcrypt via pwdlib BcryptHasher; never log raw passwords; acceptance criterion asserts $2 prefix (AUTH-03) |
| T-02-02 | Spoofing | JWT validation | mitigate | HS256 signature with .env secret; decode whitelists algorithm; 401 on InvalidToken/Expired (AUTH-04) |
| T-02-03 | Elevation of Privilege | registration peran | mitigate | RegistrasiRequest omits peran; router hardcodes peran=mahasiswa, ignores any client role (RESEARCH Security Domain) |
| T-02-04 | Tampering | SQLAlchemy queries (login/register) | mitigate | ORM parameterized queries; email lookup via filter_by, no raw SQL |
| T-02-05 | Denial of Service | /masuk brute force | accept | No rate-limiting requirement for this course-project phase (RESEARCH Security Domain — accepted risk, revisit if needed) |
</threat_model>
